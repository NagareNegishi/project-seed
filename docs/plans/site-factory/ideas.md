# site-factory: idea pool

Maturity: 🤖 ai-audited(fable-5) — generated by an expansion agent 2026-07-17,
curated by the session manager; no human review yet.

Nothing here is committed scope. The non-goals in [product.md](product.md)
stay fixed; ideas that touch one say so. Tags: cost S/M/L; when = phase from
product.md's phasing, or "later". Ranked by value for cost within each section.

## 1. Missing patterns and sections

Candidates beyond the v1 catalog cut in [catalog.md](catalog.md).

1. **SEO pack: meta tags + LocalBusiness JSON-LD.** (S, Phase 1)
   One component emitting title/description/canonical/OG tags and a
   LocalBusiness schema block (name, address, geo, hours, vertical subtype)
   from site data. Small businesses buy sites to be found on Google; this is
   the cheapest client-visible win and pure template work.

2. **Opening hours as a single data source.** (S, Phase 1)
   Hours defined once, rendered in footer, contact section, and the
   LocalBusiness schema, optionally with an "open now" badge. Kills the
   classic bug of three contradicting hour listings.

3. **404 and error pages in every template.** (S, Phase 1)
   Themed 404 with a way home, so no site ever shows a host default page.

4. **Favicon and OG-image generation.** (S, Phase 1)
   Script that takes the client logo and theme tokens and emits the favicon
   set plus a branded OG image. Tabs and shared links look finished without a
   design tool per client.

5. **Reviews / social-proof embeds.** (S, Phase 1)
   Google reviews as embed or build-time snapshot, beside testimonials.
   Third-party reviews convert better than self-written quotes, and clients
   ask for this by name.

6. **Consent-gated embeds.** (M, Phase 1)
   The catalog leans on embeds (map, scheduling, analytics), which is exactly
   what consent rules bite. A consent component that lazy-loads embeds after
   accept solves compliance and improves Lighthouse scores at once, because
   embeds stop loading on first paint. Also a fix demanded by the ops review
   (see [operations.md](operations.md)).

7. **Analytics wired at template level.** (S, Phase 1)
   Privacy-light default (Plausible/GoatCounter class) as a template slot,
   cookie-free where possible so it sits outside the consent gate. Feeds the
   care-plan report idea below, so it pays twice.

8. **Image pipeline: gallery lightbox + build-time optimization.** (M, Phase 1)
   Client photos arrive as 8 MB phone shots; resize/format/lazy-load applied
   to every image is what keeps the static-first performance promise real.

9. **Menu / price list pattern.** (S, Phase 2)
   Data-driven menu with sections, items, prices, dietary tags. Restaurants,
   salons, clinics. Key enabler for the vertical kits below.

10. **Before/after slider.** (S, Phase 2)
    Draggable comparison, no library needed. Strong seller for trades,
    dentists, cleaners, landscapers.

11. **Staff/team section with per-person booking link.** (S, Phase 2)
    Common ask for clinics and salons; composes with the booking pattern.

12. **Blog/news as a markdown collection.** (M, Phase 2)
    Operator publishes, client emails content, so the no-CMS non-goal holds.
    Near-free on Astro, and it creates recurring "post this for me" work that
    feeds the care plan.

Considered and cut: multi-language (L — real cost in every pattern and every
content slot; build only when a client pays for it) and job listings (S but no
demand signal; add on first request).

## 2. Operator leverage

1. **`site.config` manifest per client repo.** (S, Phase 1)
   One file: client, template, theme, chosen patterns, tier, domain, deploy
   target. The keystone: scaffolding, intake checklists, handover docs, and
   the fleet board all read from it. Cheap to define now, expensive to
   retrofit.

2. **Scaffold CLI: `new-site <template> <theme> --patterns ...`.** (M, Phase 1)
   Copies the template, pulls named patterns from the library, applies theme
   tokens, writes `site.config`, initializes the repo. Day-one setup drops
   from an afternoon of copying to minutes, and the assembly process can no
   longer be done wrong.

3. **Living gallery site: patterns × themes.** (M, Phase 1)
   One deployed site rendering every pattern under every theme. Sales tool
   (clients pick instead of describing wishes), regression surface for theme
   changes, and proof that a pattern works without modification.

4. **Intake checklist generated from `site.config`.** (M, Phase 1)
   The chosen patterns imply exactly which content slots exist, so generate
   the client questionnaire from them ("3 testimonial quotes, 1 hero image at
   least 1600px, opening hours, ..."). Extends the intake design in
   [intake.md](intake.md) and removes the vaguest step in the pipeline.

5. **Lighthouse budget as a delivery gate.** (S, Phase 1)
   Script that runs Lighthouse (perf, a11y, SEO) against the built site and
   fails below thresholds. Guards the static-first promise, doubles as a
   sales number, needs no CI.

6. **Client preview deploys on a staging URL.** (S, Phase 1–2)
   `<client>.preview.<operator-domain>` per project. Feedback moves from
   screenshot email threads to a link. Note: the preview/approval step is now
   a requirement (see [operations.md](operations.md)); this is its cheap
   implementation.

7. **Screenshot sweep across viewports.** (M, Phase 2)
   Playwright walks the sitemap at three widths and drops screenshots in a
   folder. One-command visual QA; the images feed the portfolio and gallery
   site for free.

8. **Content-chase sequence.** (S, Phase 2)
   Canned nudge emails at day 3/7/14, each attaching the still-empty slots
   from the intake checklist. Waiting for content is the real schedule
   killer; automating the chase converts dead time into throughput.

## 3. Business surface

1. **Care plan as the productized recurring tier.** (S, Phase 2)
   Monthly fee covering hosting, small edits, and upkeep, as the commercial
   wrapper of Tiers 2/3. Flips the permanent per-client cost from liability
   to the main profit line; recurring revenue is what makes a solo shop
   survivable between projects. The ops review reached the same conclusion
   from the risk side ([operations.md](operations.md)).

2. **Footer credit as a referral loop.** (S, Phase 1)
   "Site by X" in every delivered footer, linking to the gallery site. Zero
   marginal cost; every delivered site advertises in exactly the neighborhood
   the next client lives in. Removable for a fee.

3. **Vertical starter kits.** (M, Phase 2)
   Named preset per niche: template + pattern selection + copy skeleton +
   schema subtype ("clinic kit", "restaurant kit", "trades kit"). Selling
   "sites for clinics" beats selling "websites"; the marginal cost is a
   preset file.

4. **Quarterly report email for care-plan clients.** (M, Phase 3)
   Generated from the analytics API plus an uptime check: visits, top pages,
   uptime, changes made. The artifact that justifies the recurring fee, and
   it is a script, not a service.

5. **Template/theme marketplace.** (L, later)
   Adjacent to the "not a public framework" non-goal and carries real support
   burden. Parked; the gallery site keeps the option alive as a
   storefront-shaped asset.

## 4. Factory evolution (inside the non-goals)

1. **Template as git upstream remote in every client repo.** (S, Phase 1)
   No automatic propagation, so the non-goal holds, but cherry-picking a
   pattern fix into a client site becomes a one-command, billable act instead
   of a diff hunt.

2. **Drift ledger.** (S, Phase 1)
   Generated file per client repo listing copied pattern versions (name +
   library commit). When a bug is found in a pattern, grep the fleet for who
   has it. Accepting drift is fine; accepting untracked drift is not.

3. **Fleet board.** (M, Phase 2)
   One private page listing every delivered site with tier, uptime ping, TLS
   and domain expiry, last Lighthouse score, driven by `site.config`. A solo
   operator cannot remember twenty renewal dates; this is the difference
   between running 5 sites and 30.

4. **Config-driven assembly, repo-shaped output.** (M, Phase 2)
   Grow the scaffold CLI until `site.config` fully determines the copied
   files. Output stays a plain repo, so this is not the parked engine, but it
   captures most of the engine's leverage and is the stepping stone if that
   non-goal is ever reopened at volume.

5. **Which non-goal to revisit first: no client CMS.** (L, later)
   A git-backed editor (Decap/PagesCMS class) over markdown content keeps
   site-as-repo intact. It is the non-goal whose removal changes revenue
   most: care-plan clients who edit their own hours and menus stop emailing
   the operator, which raises how many care plans one person can hold.
   Revisit after the care plan has ~10 clients, not before.
