# v1 — allow git reads and scoped add

Designed 2026-07-12, applied to `.claude/settings.json` 2026-07-12. Extends [v0](v0-baseline-strict.md) in two ways: allow rules for read-only git commands, and a three-tier policy for `git add`. Commit stays at ask, push and the rest of the deny list stay denied.

## Intent

**Reads:** let Claude inspect history and the remote's published state without a prompt per command. The trigger was the pr-draft skill: with `git fetch` denied, it drafted from a stale local base and described a wider changeset than the actual PR. `git ls-remote` lets it detect that staleness (compare the remote's reported SHA against the local remote-tracking ref) without writing anything, not even remote-tracking refs.

**Add:** every file Claude edits already passes review at write time (Edit/Write prompt per file), so prompting again for `git add <explicit path>` is a duplicate review. The prompt that matters is `git commit`, which stays at ask as the sign-off for what enters history. Bulk staging is different: it could sweep in files that never passed a review prompt (files generated by Bash commands, the user's own working-tree edits, pre-existing untracked files), so every bulk form is hard-denied.

Rules of engagement, stated as intent because permission rules cannot see session history:

- Stage only explicit paths of files worked on in the session. Never bulk-stage.
- Show `git status` and `git diff --stat --staged` before requesting any commit, so the commit prompt is an informed sign-off.

Known gap: no rule can stop staging an explicit path of a file that was not worked on this session. The commit ask-gate is the check for that.

## What was added

Reads (allow):

| Rule | Covers | Why |
|---|---|---|
| `Bash(git log *)` | commit history | drafting PR text and release notes from history |
| `Bash(git show *)` | commits, tags, file contents at a ref | inspecting a specific commit while drafting |
| `Bash(git rev-parse *)` | resolving refs and SHAs, repo paths | base-branch detection (`--abbrev-ref refs/remotes/origin/HEAD`) and SHA comparison |
| `Bash(git ls-remote *)` | network read of a remote's refs and SHAs | staleness check for pr-draft; verifying a branch or tag exists on the remote. Writes nothing locally |

Add (deny is evaluated before allow, so the narrow rules fire even though `git add *` also matches — see [Configure permissions](https://code.claude.com/docs/en/permissions)):

| Tier | Rule | Why |
|---|---|---|
| deny | `Bash(git add .)`, `Bash(git add . *)`, `Bash(git add ./)` | current-directory catch-all |
| deny | `Bash(git add -A)`, `Bash(git add -A *)`, `Bash(git add --all)`, `Bash(git add --all *)` | stage-everything forms |
| deny | `Bash(git add -u)`, `Bash(git add -u *)`, `Bash(git add --update)`, `Bash(git add --update *)` | all-tracked-changes forms |
| deny | `Bash(git add -f *)`, `Bash(git add --force *)` | stages gitignored files, which are ignored for a reason (build output, secrets). If one ever belongs in the repo, the user un-ignores and stages it themselves |
| allow | `Bash(git add *)` | explicit-path staging without a duplicate prompt |

A single `*` spans spaces, so `git add -A src` is caught by `git add -A *`. `git add *` moved from ask to allow; `git commit *` remains in ask.

## Deliberate omissions

- `git symbolic-ref` — not allowed even though pr-draft used it for base detection. Its two-argument form writes refs, and a prefix rule cannot separate read from write use. Decision: edit pr-draft to use `git rev-parse --abbrev-ref refs/remotes/origin/HEAD` instead.
- `git branch --list` and `git stash list` — not allowed. The deny rules `git branch *` and `git stash *` match first, so an allow entry would never fire, and narrowing the denies would give up the hard block on branch mutation. When a flow needs branch names, `git for-each-ref refs/heads` gives the same list read-only.
- `git remote -v` — not added in this version; no flow asked for it yet.
- Flag-after-path spellings such as `git add file -f` are not covered by the add patterns; prefix matching cannot reorder arguments. Covered by the explicit-path intent and the commit gate.
- Shell globs (`git add src/*`) and directory paths (`git add src/`) are not denied; they cannot be textually separated from explicit files. Same fallback.
- Unstaging stays prompted: `git reset` is denied, so unstaging uses `git restore --staged <path>`, which is unlisted and prompts.
- No fetch, no push, no changes to the rest of ask or deny.

## Changed from v0

- `allow`: added `git log *`, `git show *`, `git rev-parse *`, `git ls-remote *`, `git add *`.
- `ask`: removed `git add *`; `git commit *` unchanged.
- `deny`: added thirteen `git add` patterns (bulk and force forms). Everything else is byte-identical to v0.
